# Phase 2 — 技术拆解与项目初始化

---

## 架构决策检查点（所有模式均需执行，在任务拆解之前）

在拆解任务之前，必须先确定以下架构决策，记录到 `PROJECT_CONTEXT.md`。
**未完成决策不得进入任务拆解阶段。**

对每项决策：用户有明确偏好则按用户意见；**用户无偏好时，Tech Lead 给出推荐方案并说明理由，直接采用推荐值继续，不阻塞流程。**

```
架构决策确认：
□ 认证方案：[JWT / Session / 无认证] + Token 存储方式
□ API 设计规范：[endpoint 命名约定 / 统一错误格式 / 分页结构]
□ 数据库主要表结构：[核心表和字段（草稿级）]
□ 数据库迁移：[是否有存量数据？有 → 必须指定迁移框架（Python: Alembic / Node: Knex / Java: Flyway）]
□ 前后端分离项目：是否需要生成 API Contract 文档（OpenAPI / 接口清单）
□ 代码风格约定：[命名规范 / 目录结构约定]

以上决策向用户确认后，方可进入任务拆解。
```

**架构决策完成后立即更新 PROJECT_CONTEXT.md，不等到 Phase 5。**

## 完整模式（新项目 / 新功能）

### 架构级变更时：先执行变更影响评估

当需求与 PROJECT_CONTEXT.md 中已有架构决策冲突时（如替换认证方案、重写权限系统），**在任务拆解之前**必须先执行：

1. 列出受影响的已合并 PR（用 `gh pr list --state merged` 查找与冲突模块相关的 PR）
2. 为每个受影响的已合并 PR 创建修复 Issue（标记：`[架构变更] 修复受 #N PR 影响的代码`）
3. 检查所有 open Issue，关闭或修订与新架构冲突的 Issue（在 Issue 评论中说明原因）
4. 立即更新 PROJECT_CONTEXT.md 的架构决策部分（不等到 Phase 5）

### 执行步骤

1. 基于已确认的架构决策，将需求拆解为独立可并行的开发任务：
   - 可由单个 Agent 独立完成
   - 有明确的输入、输出、完成标准
   - 与其他任务的依赖关系清晰
   - **识别跨任务共享的基础设施组件**（数据库连接层、认证中间件、公共 utils、API Client 封装），为每个组件单独创建 Issue，标记为其他任务的前置依赖

2. **新项目**时在 GitHub 上执行：
   - `gh repo create [项目名] --private` 创建仓库
   - 将 PRD 内容作为 Issue #1 创建（标题：`[PRD] 产品需求文档`）
   - 在仓库根目录创建 `PROJECT_CONTEXT.md`（参考 `~/.claude/commands/dev/PROJECT_CONTEXT_TEMPLATE.md` 中的格式）
   - 如需 API Contract：创建 `API_CONTRACT.md`，记录所有 endpoint 定义，作为前后端 Issue 的共同约束
   - 为每个开发任务创建对应 Issue（使用下方 Issue 模板）
   - 创建 milestone 关联所有 Issue

3. **已有项目**时：
   - 读取 `PROJECT_CONTEXT.md` 恢复上下文
   - 为新需求创建对应 Issue（使用下方 Issue 模板）
   - 更新 milestone

4. 展示任务清单给用户确认，使用显式依赖格式：
   ```
   任务清单（共 N 个）：

   [基础设施层 — 必须先完成，其他任务依赖它]
   □ Issue #1 [基础设施任务] — 产出：xxx — 阻塞：#3, #4, #5

   [可并行开发层]
   □ Issue #3 [功能A] — 产出：xxx — 依赖：#1 — 可与 #4 并行
   □ Issue #4 [功能B] — 产出：xxx — 依赖：#1 — 可与 #3 并行

   [收口层 — 所有功能完成后]
   □ Issue #6 [集成/联调] — 依赖：#3, #4, #5
   ```

5. **用户确认优先级/顺序后进入 Phase 3**

## 极速模式（紧急 Hotfix）

**跳过架构决策检查点，跳过 QA（Phase 3.5）。**

1. 直接创建一个 Hotfix Issue，标题格式：`[Hotfix] [故障描述]`
2. 验收标准只需覆盖：故障复现路径 + 修复验证
3. 展示 Issue 给用户确认后，**立即进入 Phase 3（单 Agent，使用 `worker-fix.md`）**
4. PR 合并后**必须执行受影响 PR 协调**（见 phase4.md 合并顺序章节）

---

## 轻量模式（小改动 / Bug 修复）

1. 直接创建一个 Issue（使用下方 Issue 模板）
2. 不需要任务拆解和 milestone
3. 展示 Issue 内容给用户确认后，**立即进入 Phase 3（单 Agent）**

---

## 重构模式

重构任务必须满足：
1. Issue 验收标准使用**双维度格式**：
   - 结构指标：`[文件/模块] 行数/依赖/复杂度指标 → 目标值`（例如：`utils.py 行数 → 不超过 200 行`）
   - 回归指标：`全量测试通过率 → 100%，无新增 lint 错误`
2. 重构 Issue **必须放入基础设施层**，所有依赖该模块的新功能 Issue 标记为依赖它，**不得并行**；不依赖被重构模块的其他功能 Issue 可以同时并行开始
3. Worker Agent 使用 `worker-fix.md`，自检时重点验证：所有原有调用方无破坏性变更

## Issue 模板

```markdown
## 任务描述
[背景和目标]

## 验收标准（工程可验证格式）
每条必须按以下结构：[触发条件] → [期望响应/状态/副作用]
示例：POST /auth/register 传入已存在邮箱 → 返回 409，body 包含 error_code

- [ ] [触发条件1] → [期望结果1]
- [ ] [触发条件2] → [期望结果2]
- [ ] [边界情况：空输入/极值] → [期望结果]
- [ ] [如有测试框架：全量测试通过，无回归]

## 架构约束引用
[引用 PROJECT_CONTEXT.md 或 API_CONTRACT.md 中的相关决策]

## 技术要点
[关键实现约束或注意事项]

## 不做的事
[明确排除项，避免 scope creep]
```
