# Phase 1 — 产品对齐

不消除所有模糊点，不进入开发。

---

## 判断用户文档类型

**情况 A — 用户已提供需求文档**

最低门槛（三项都满足才算情况 A，否则进入情况 B）：
- 有核心功能描述（不只是一句话的产品方向）
- 有目标用户或使用场景
- 有技术约束或明确的非功能要求

执行步骤：
1. 直接以用户提供的文档为基准，**不重新生成 PRD**
2. 进行**开发影响性歧义检查**，一个歧义点满足以下任意一条才值得提问：
   - 不同解读会导致数据库 schema 不同
   - 不同解读会导致 UI 交互流程不同
   - 不同解读会导致某个模块存在与否（系统边界不同）
   - 不同解读会导致技术方案选型不同
3. **最多提 5 个问题**，按重要性排序，格式：
   ```
   基于你的需求文档，有以下 N 个关键点需要确认：
   1. [具体问题] — 影响：[哪个具体开发决策]
   ```
4. 用户回答后，以原始文档 + 回答作为开发依据，进入 Step 2 确认
5. **不得对用户文档进行"重写"或"规范化"**，除非用户明确要求

---

**情况 B — 用户未提供需求文档**

最多两轮追问：

**第一轮（最多 3 个，核心决策性问题）：**
先问清楚决定产品性质的核心问题，通常是：
- 核心能力是什么（产品最本质的事）
- 目标用户是谁、典型使用场景
- 部署方式（Web / 桌面 / API / 本地）

基于回答，决定是否需要第二轮。

**第二轮（最多 3 个，仅当第一轮引发新歧义时）：**
- 哪些功能是第一版（MVP）必须有的
- 技术约束
- 成功标准

两轮完成后输出 PRD，请用户确认（格式见下方）。

---

## Step 2 确认（情况 A 和 B 均需执行）

输出确认摘要，用户明确确认后才能进入 Phase 2：

```
我将基于以下理解进入开发：
- 核心功能：[列出]
- MVP 范围（第一版做）：[列出]
- 明确不做的事：[列出]
- 技术约束：[列出]

请确认（回复"确认"或指出需要调整的地方）。
```

---

## PRD 输出格式（仅情况 B 时生成）

```
## 项目名称
## 目标（一句话）
## 用户与场景
## MVP 功能（第一版必须有）
## V2 功能（后续迭代考虑）
## 不做的事（明确排除项）
## 成功标准
## 技术约束
```

---

## 禁止行为

- 不得在 Step 2 确认完成前进入开发
- 不得假设任何未明确的需求
- 不得超过两轮追问
- 不得在用户已有文档的情况下重新生成 PRD
