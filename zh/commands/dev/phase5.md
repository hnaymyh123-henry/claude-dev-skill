# Phase 5 — Retro & 技术债清扫

所有 PR 合并后自动触发，分两步执行：先 Retro，再技术债清扫。

---

## Step 1 — Retro

输出格式：

```
## Retro — [项目名] / [迭代描述]
### 已完成
### 已知问题
### 未完成（推迟到下一轮）
### 建议下一轮优先级
```

更新 `PROJECT_CONTEXT.md` 当前状态部分 + `docs/feature-log.md` 已完成列表。

---

## Step 2 — 技术债清扫（不得跳过）

**扫描范围：本轮 PR 涉及的目录及其上下游模块。不做全库扫描，聚焦改动区域。**

### 2a — 死文档清查

```bash
# 列出 docs/ 下最近 N 次 commit 未被修改的文件
git log --since="上轮迭代开始日期" --name-only --pretty=format: -- docs/ | sort -u
```

对每个疑似死文档：
- 检查它引用的功能/模块是否仍然存在
- 引用对象已删除 → **直接删除该文档**，`git rm`
- 内容严重过时但结构仍有用 → 更新内容，注明 `最后验证：YYYY-MM-DD`
- 判断不确定 → 在 `docs/feature-log.md` 技术债部分登记，**不要保留未经验证的猜测**

### 2b — 废弃代码清查

对本轮 diff 涉及的文件，检查以下 4 类：

```
□ 注释掉的代码块（连续 3 行以上的注释代码）→ 直接删除
□ 未被任何地方引用的函数/类（用 grep 验证）→ 直接删除
□ 已合并的 feature flag（值固定为 True/False 的条件分支）→ 内联删除
□ TODO/FIXME 注释 → 检查是否已解决：已解决删除注释，未解决创建 Issue 并删除注释
```

验证方式：
```bash
# 查找未引用的函数（Python 示例）
grep -rn "def function_name" src/
# 验证调用方
grep -rn "function_name(" src/ tests/
```

### 2c — 失效功能登记

不在本轮清扫范围内但已知失效的功能（如旧版 API endpoint、废弃的 webhook）：
- 在 `docs/feature-log.md` 技术债部分登记
- 格式：`[描述]（来源：PR #N，登记时间：YYYY-MM-DD，清扫目标迭代：下一轮/待定）`
- **不要就地删除未经本轮测试的代码**，留给专项重构 Issue 处理

---

## Step 3 — 清扫结果输出

```
## 技术债清扫报告 — [迭代描述]

### 已删除
- [文件/函数/代码块]：[原因]

### 已更新
- [文档名]：[更新内容摘要]

### 新登记技术债（待下轮处理）
- [描述]（预计影响：低/中/高）

### 跳过（原因）
- [如有，说明为什么某项未清扫]
```

---

## Step 4 — 循环

Retro + 清扫完成后输出：

```
项目当前处于待命状态。你可以：
- 提出新需求（我会自动分类并走对应流程）
- 说"先停这里"结束本轮开发
```

用户提出新需求 → 回到 Phase 0。
